# PSQL.md — Migrating `ocms` from MariaDB to PostgreSQL

> **Goal:** move the cleaned, keyed schema (per [ANALYSIS.md](ANALYSIS.md) +
> [RELATIONSHIP.md](RELATIONSHIP.md)) from MariaDB/InnoDB to PostgreSQL **without data loss**.
> This document lists every concrete blocker found in the data, the type mapping, and a
> step-by-step runbook.

**Source:** MariaDB, InnoDB, 559 tables, ~23 M rows, no FKs, `utf8mb4` assumed.
**Recommended tool:** [`pgloader`](https://pgloader.io) (purpose-built for MySQL→PostgreSQL;
handles types, sequences, and zero-dates in one pass), with a hand-written cast/exclusion file.

---

## 1. Migration blockers found in *this* database

These are not generic warnings — each was measured against the live data.

### 🔴 B1. Zero-dates (`0000-00-00`) — **must fix or load fails**
MariaDB allows `0000-00-00`; PostgreSQL rejects it. They are widespread here:

| Table.column | Zero-date rows |
|---|---:|
| `dyeing_processStatus.statusTime` | 112 016 |
| `dyeing_batchCard.shipDate` | 21 119 |
| `knittingYarnAllocation.shipDate` | 16 839 |
| `inventory_grayFabricReceiveDetails.shipDate` | 13 517 |
| `workorder_knittingDetails.shipDate` | 13 266 |
| `inventory_grayFabricIssueDetails.shipDate` | 11 742 |
| `ocms_chemical_transaction.tr_date` | 1 038 |
| …many more `shipDate` / `issueDate` columns | |

**Fix:** convert `'0000-00-00'` → `NULL` on load. `pgloader` does this automatically; if using
`mysqldump` set the source session `sql_mode` / use a converting export. **This is the #1 thing
that will break a naive migration.**

### 🔴 B2. Out-of-range / garbage dates
Recency scan found `MAX(date)` of **year 3023** and **2030** in some tables (data-entry typos).
PostgreSQL `date` accepts these, but they corrupt reports. Quarantine/correct during load
(flag rows where `date > CURRENT_DATE + interval '1 year'`).

### 🟠 B3. Mixed-case identifiers — **283 of 559 tables**
MariaDB on Linux is case-insensitive-ish for column refs and folds table names per
`lower_case_table_names`; PostgreSQL folds **unquoted** identifiers to lower case and treats
quoted ones as case-sensitive. Names like `dbProjectInfo`, `inventory_DyesChemIssueDetailsSub`,
`libCountry` would each then require `"…"` quoting forever.
**Recommendation:** **rename everything to `snake_case`** during migration
(`dbProjectInfo`→`db_project_info`, `inventory_DyesChemIssueDetailsSub`→
`inventory_dyes_chem_issue_details_sub`). Do it via a mapping table so app code can be updated
mechanically. (Alternative: quote all identifiers — not recommended, it's a permanent tax.)
> Note: no two table names collide when lower-cased (verified), so a clean snake_case rename is
> safe. Watch the near-duplicates being merged anyway: `libCountry`/`libcountry`,
> `libsupplier`/`lib_supplier` — resolve in [ANALYSIS.md §4](ANALYSIS.md) *before* renaming.

### 🟠 B4. `double` used for money — 1 256 `double` columns
Prices/amounts (`unitPrice` appears in 118 tables, plus `value`, `amount`, `qty`…) are stored
as `double`, which is lossy for currency. Migrate money columns to `numeric(18,4)` and qty to
`numeric` where exactness matters. (Loading `double`→`numeric` is lossless in direction; pick
scale per column.)

### 🟡 B5. `tinyint(1)` booleans — 98 `tinyint` columns
Flags like `isConfirm` (170 tables), `sendApproval` (85), `status`, `deleted` are `tinyint`.
PostgreSQL has real `boolean`. Map `tinyint(1)` flag columns → `boolean` (0/1 → false/true);
keep wider `tinyint` as `smallint`.

### 🟡 B6. `enum` — 13 columns
MySQL `ENUM` has no direct PG equivalent. Convert to `text` + `CHECK (col IN (...))`, or a
native `CREATE TYPE … AS ENUM`. `text + CHECK` is easier to evolve.

### 🟡 B7. AUTO_INCREMENT — 974 columns
Map to `GENERATED BY DEFAULT AS IDENTITY` (or `bigint` + sequence). After load, **reset every
sequence** to `MAX(id)+1` (pgloader does this; verify).

## 2. Type mapping (MariaDB → PostgreSQL)

Observed source-type counts → target:

| MariaDB type | Count | PostgreSQL target | Notes |
|---|---:|---|---|
| `int` | 5 264 | `integer` (`bigint` if values near 2³¹) | ids → consider `bigint` for growth |
| `varchar` | 2 880 | `varchar(n)` / `text` | keep length or just use `text` |
| `double` | 1 256 | `double precision`, **`numeric(18,4)` for money** | see B4 |
| `date` | 1 125 | `date` | apply B1/B2 cleaning |
| `text` | 375 | `text` | |
| `datetime` | 355 | `timestamp` (`timestamptz` if TZ-aware) | |
| `tinyint` | 98 | `boolean` (flags) / `smallint` | see B5 |
| `bigint` | 20 | `bigint` | |
| `float` | 13 | `real` / `numeric` | |
| `enum` | 13 | `text` + `CHECK` | see B6 |
| `longtext` | 7 | `text` | |
| `smallint` | 6 | `smallint` | |
| `char` | 6 | `char(n)` | |
| `timestamp` | 3 | `timestamptz` | watch default `CURRENT_TIMESTAMP` |
| `decimal` | 1 | `numeric` | |

Other syntax: backticks `` `x` `` → double quotes `"x"`; `` `db`.`tbl` `` → `schema.tbl`;
`UNSIGNED` → drop (use `bigint`/`CHECK >= 0`); `ON UPDATE CURRENT_TIMESTAMP` → a trigger.

## 3. Runbook

### Phase 0 — Pre-clean in MariaDB (lossless, reversible)
1. Apply [ANALYSIS.md](ANALYSIS.md): drop empty/abandoned tables (after dumping to archive),
   merge the duplicate masters (`lib_supplier`, `lib_yarn_type`, `libbuyer`, `dbProjectInfo`).
2. Apply [RELATIONSHIP.md §4 Steps 1–4](RELATIONSHIP.md): add `UNIQUE` on business keys, add
   enforceable FKs, quarantine orphans into `*_orphans` tables (no deletes).
3. Snapshot/backup the source DB.

### Phase 1 — Schema move with `pgloader`
Use a load file that renames to snake_case, casts types, and nulls zero-dates:
```lisp
LOAD DATABASE
     FROM mysql://sibyz@localhost/ocms
     INTO postgresql://sibyz@localhost/ocms

 WITH include drop, create tables, create indexes, reset sequences,
      downcase identifiers                 -- B3: fold names to lower/snake-ish

 SET work_mem to '256MB', maintenance_work_mem to '1GB'

 CAST type datetime  to timestamptz drop default drop not null using zero-dates-to-null,
      type date      to date        using zero-dates-to-null,           -- B1
      type tinyint   when (= precision 1) to boolean using tinyint-to-boolean,  -- B5
      type double    to double precision

 EXCLUDING TABLE NAMES MATCHING ~/_collar$/, 'knit_qad', 'workorder_yarn'  -- B/dead tables

 BEFORE LOAD DO $$ CREATE SCHEMA IF NOT EXISTS ocms; $$;
```
> `pgloader` ships the `zero-dates-to-null` and `tinyint-to-boolean` transforms. Money columns
> (B4) and `enum`→`CHECK` (B6) are best done by pre-defining target DDL and loading data only,
> or a post-load `ALTER … TYPE numeric USING …`.

### Phase 2 — Constraints & integrity (in PostgreSQL)
4. Re-create / verify PKs and the `UNIQUE` business keys.
5. Add the FKs from [RELATIONSHIP.md §2](RELATIONSHIP.md); use `NOT VALID` then `VALIDATE` for
   the 70–95 % edges.
6. Convert money `double`→`numeric` (B4), enums→`CHECK` (B6).
7. `ANALYZE;` the whole DB; rebuild indexes that aid the big tables (`activity_log`,
   `production_sewingInfoDetails`, `ocms_doc_approval_sts`).

### Phase 3 — Verify (the lossless proof)
8. **Row-count parity:** compare `COUNT(*)` per table MariaDB vs PostgreSQL (should match
   exactly, minus intentionally-dropped tables).
9. **Checksum parity** on key tables: compare ordered hash of business columns.
10. **Spot-check** the dates that were zero/garbage now read `NULL`/corrected.
11. Point a read replica of the app at PG; run reports; diff against MariaDB output.

## 4. Effort & risk summary

| Item | Risk | Effort |
|---|---|---|
| B1 zero-dates | High (hard load failure) | Low (pgloader transform) |
| B3 283 mixed-case renames | Medium (app code churn) | Medium (mechanical, scriptable) |
| FK adoption + orphan cleanup | Medium (business decisions) | Medium–High |
| B4 money precision | Medium (silent rounding today) | Medium |
| B5/B6/B7 booleans/enums/sequences | Low | Low |
| Dead-table pruning | Low | Low |

**Recommended sequence:** Phase 0 cleanup in MariaDB first (so the move carries a clean, keyed,
deduped schema), then `pgloader`, then constraints, then verify. Do it on a copy; keep the
MariaDB source until row-count + checksum parity passes.
